# 設計思想

## この文書について

本書は、AI経費処理システムの設計思想を定義する。

本プロジェクトにおける設計・実装・保守は、本書を最上位の設計指針とする。

システムの機能、利用するAI、開発基盤、データ保存先が変わっても、本書に記載する設計思想は原則として変更しない。

---

# Mission（使命）

AIは人を置き換えるためではなく、人がより価値の高い仕事に集中できるよう支援するために存在する。

AI経費処理システムは、経費処理業務を効率化し、入力・確認・判断を支援することを目的とする。

---

# Vision（目指す姿）

本システムは、領収書・請求書・カード明細などの経費関連情報をAIが理解し、共通経費データとして管理する基盤となることを目指す。

また、特定の会計ソフトや実装技術に依存せず、さまざまな会計システムや将来のプラットフォームへ展開できる業務支援システムとして発展させる。

---

# 設計原則

## 設計原則1

### 業務モデルを中心に設計する。

本システムは、特定の技術やプラットフォームを前提に設計しない。

Google Apps Script、Google Sheets、Gemini API などは実装手段であり、システムの本質ではない。

設計は、経費処理・経費管理・経費チェックという業務モデルを中心に行う。

---

## 設計原則2

### AIは会社固有ルールを保持しない。

会社ごとに異なる勘定科目、経費区分、承認ルール、会計処理ルールは設定情報として保持する。

AIプロンプトやプログラムへ固定値として埋め込まない。

---

## 設計原則3

### 会計ソフトへ依存しない。

システム内部では共通経費データを保持する。

会計ソフト固有の仕様、勘定科目コード、税区分、出力形式は Exporter が吸収する。

---

## 設計原則4

### 共通データモデルを中心に設計する。

本システムは、会計ソフト別データではなく共通経費データを中心に設計する。

各会計ソフト向けデータは、共通経費データから生成する。

---

## 設計原則5

### AIは判断を支援し、人が最終確認を行う。

AIは可能な限り自動判定を行う。

ただし、重要な判断については利用者が確認・修正できる仕組みを基本とする。

---

## 設計原則6

### 経費を入力するだけでなく、継続的にチェックする。

本システムは、領収書やカード明細を入力するだけのシステムではない。

重複登録、登録漏れ、継続課金、料金変動、不要な契約、通常と異なる支出などを検知し、経費処理の品質向上とコスト削減を支援する。

---

## 設計原則7

### 保守性を最優先する。

短期的な実装速度よりも、将来の保守・機能追加・会計ソフト追加・プラットフォーム変更への対応を容易にする設計を優先する。

---

## 設計原則8

### コードより設計を優先する。

新しい機能を追加する場合は、次の順番で進める。

1. 設計書更新
2. 実装
3. テスト
4. リリース

---

# 開発方針

* GitHubをソースコードと設計書の正本とする。
* Apps Script エディタは原則として直接編集しない。
* VS Code を標準の開発環境とする。
* 仕様変更時は、コードより先に docs を更新する。
* 要件定義・設計書では、実装技術に依存しない表現を優先する。
* 実装技術に関する内容は、実装構成または開発ルールに分離する。
* 会計ソフト固有仕様は、共通設計へ持ち込まない。

---

# 本書の位置付け

本書は、AI経費処理システムだけでなく、今後開発する CSplus のAI業務支援システム全体で共通して適用する設計思想とする。
